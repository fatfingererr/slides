Preprint 2025

为什么 Harness 工程如此重要？

  • Harness 定义了模型看到什么信息——其重要性不亚于模型权重本身
  • 改变 Harness 可使同一模型在相同基准上产生 6 倍性能差距
  • 现有 Harness 仍主要依赖人工设计，迭代效率低下
  • 现有文本优化方法存在根本性局限：过度压缩反馈
核心洞察：Harness 工程需要一个能够"理解"并利用完整执行历史的优化器，而非仅从标量分数或简短摘要中学习。

为什么选择 Coding Agent 作为 Proposer？

  • 上下文超限：经验数据迅速超过上下文限制，需主动选择查看什么
  • 原始代码诊断：可直接浏览先前候选的源代码，而非依赖摘要
  • 执行轨迹分析：能够追溯失败案例的具体执行步骤
  • 代码编辑验证：通过直接与代码库交互验证修改

Meta-Harness 的关键创新

  • 文件系统作为记忆：存储所有候选的源代码、分数和执行轨迹
  • 选择性诊断：Coding Agent 决定查看什么，而非被动接收压缩摘要
  • 端到端搜索：在代码空间中搜索，小的改动可能多步后产生大影响
  • 最小外层设计：不依赖预定义搜索结构，随着 Coding Agent 能力提升而自动改进
  • 原始轨迹访问：通过失败案例和执行轨迹追溯失败根因
范式转变：从"优化文本摘要"到"让 Agent 自主探索完整经验数据"

结论与未来展望

  • Harness 工程可以完全自动化，且自动化发现的 Harness 超越手工设计
  • 丰富的历史经验访问（vs. 压缩反馈）是提升优化效率的关键
  • 随着 Coding Agent 能力持续进步，Meta-Harness 将自动受益
  • 为未来"元优化器"的设计提供了新范式
核心洞察：当优化目标是完整可执行程序，且环境反馈分散在代码、分数和执行轨迹中时，让优化器直接"阅读"原始数据比提供人工摘要更有效。
