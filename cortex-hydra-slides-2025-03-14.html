傳統 RAG 的兩大致命傷

Retrieval-Augmented Generation（檢索增強生成）為何無法勝任長期記憶？

🔨 語義碎片化

Semantic Fragmentation

簡單的「切塊」策略會破壞事實之間的關聯

舉例：
區塊 A：「我討厭那個框架」
區塊 B：「React 很難除錯」

→ 系統無法連結「那個框架」=「React」

⏰ 時間停滯

Temporal Stagnation

向量資料庫無法區分「現在住哪」和「以前住哪」

舉例：
2022年：「我住在紐約」
2024年：「我搬到倫敦了」

→ 系統可能回答「你住在紐約」

核心技術 (1)：Sliding Window Inference Pipeline

滑動窗口推導管道 — 讓每個記憶片段都「自包含」

解決什麼問題？

孤立的代名詞
當使用者說「那個框架很難用」，系統需要知道「那個框架」指的是什麼

實體解析 讓「它」、「那個」、「他」都不再是謎

如何運作？

  • 每個片段 sᵢ 都有一個「前後文窗口」Wᵢ
  • 包含前面 hₚᵣₑᵥ 段和後面 hₙₑₓₜ 段
  • 使用輕量級語言模型進行轉換
輸入：「我討厭它，除錯很麻煩」

輸出：「使用者 (前端工程師) 討厭 React，因為除錯很麻煩」

核心技術 (2)：Git-Style 版本化知識圖譜

像 Git 管理程式碼一樣管理記憶 — 永不遺失歷史

關鍵洞察：當使用者「改變想法」時，我們不應該「覆蓋」舊資料，而應該「新增」一筆變更記錄

傳統做法

❌ 直接更新：「紐約」→「倫敦」

舊資訊被覆蓋，失去時間線和變更原因

Hydra DB 做法

✅ Append-Only（僅追加）

2022: 住在紐約（因為在 XYZ 新創工作）
2024: 搬到倫敦（因為轉職 Meta，想離父母近一點）

這樣就能回答：「你為什麼搬家？」、「2023年你住哪？」

關鍵啟示與學習重點

從 Hydra DB 設計中學到的架構思維

核心觀點：長期記憶的品質取決於「預處理和表徵設計」，而非「原始模型能力」
  • 不要只是儲存文字 — 儲存實體、關係、時間線
  • 避免破壞性更新 — 使用 Append-Only 保留歷史脈絡
  • 多層次檢索 — 結合向量相似度與圖譜結構
  • 自包含片段 — 每個記憶都應能獨立被理解
  • 模型無關性 — 好的記憶層應適用於各種規模的 LLM
💡 給學習者的建議：
當設計 AI 系統時，思考「資訊如何在不同時間點被連結和檢索」， 比單純追求「更大的上下文窗口」更重要。
