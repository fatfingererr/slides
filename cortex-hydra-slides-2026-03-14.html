Hydra DB

Beyond Context Windows for Long-Term Agentic Memory

突破上下文視窗限制，打造 AI Agent 的長期記憶系統

📄 論文解讀 🎯 適合對象：資訊相關科系大學生 📅 2026.03.14

01 / Background

AI 從單輪對話邁向多輪 Agent 系統

🔹 過去：單輪對話
每次對話獨立處理，無需考慮歷史脈絡。就像每次見面都重新自我介紹，AI 無法記得你上次的偏好。
🔹 現在：Agent 系統
AI 需要執行多步驟任務，持續與環境互動。這需要長期記憶來維持上下文連貫性。

「真正的智能助理，應該像認識多年的朋友，記得你的習慣、經歷與偏好。」

02 / Problem

傳統 RAG 面臨的兩大挑戰

RAG（檢索增強生成）雖然強大，但在多輪對話場景存在根本缺陷

⚠️ 語義碎片化

  • 簡單的文本分塊（Chunking）會破壞時間順序和語義關聯
  • 「那個框架」在不同段落可能指完全不同的事物
  • 無法理解跨段落的事件因果關係

⚠️ 時間停滯

  • 無法區分過時事實與當前真相
  • 使用者說「我現在改用 Vue」，舊偏好「React」仍被檢索
  • 缺乏事實的有效時間範圍記錄

04 / Core Tech 01

Sliding Window Inference Pipeline

滑動窗口推導管道 — 確保每個記憶片段都「自包含」

🔗 實體解析（Entity Resolution）
將模糊的指代詞解析為具體實體。例如將「那個框架」解析為「React 18」，確保檢索時語義完整。
🌉 上下文橋接（Context Bridging）
在滑動窗口間建立連接，保留跨窗口的語義關聯。如同閱讀小說時，章節間的情節不會斷裂。

傳統方法：「那個框架很好用」→ 無法理解是哪個框架
Hydra DB：「那個框架（React 18）很好用」→ 語義完整可檢索

05 / Core Tech 02

Git-Style Version化知識圖譜

用「只追加、不覆蓋」的方式記錄狀態變化

📦 版本化概念

  • 類似 Git 的 commit 歷史，每個事實都有時間戳記
  • 新偏好覆蓋時，舊版本保留，不會遺失
  • 可以追溯「某人在某時認為...」

⏰ 時間感知檢索

  • 每個事實都記錄有效時間範圍
  • 檢索時優先返回「當前有效」的事實
  • 支援時間推理：「你在 2024 年之前用什麼框架？」
🌲 視覺化呈現
[2024-01] 偏好框架: React ──┐
[2024-06] 偏好框架: Vue  ────┤→ 兩個版本同時存在，可查詢任意時間點
[2024-09] 偏好框架: Svelte ──┘

06 / Core Tech 03

多維向量基底（Vector Substrate）

同時索引多種表徵，提升檢索精準度

傳統向量資料庫只存語義向量，Hydra DB 建立多層次索引：

📝 原始內容索引
完整的語義嵌入向量，捕捉深層語義關聯。適合處理「意思相近但用詞不同」的查詢。
🔤 稀疏關鍵字索引
保留關鍵字資訊（如 BM25）。適合精確匹配特定術語，如「Transformer」、「BERT」。
💭 潛在語境索引
記錄對話的上下文背景、說話者意圖、情感狀態等隱含資訊。
🎯 混合檢索優勢
三種索引互補，既能理解語義相似性，又能精確匹配關鍵詞，還能考慮上下文脈絡。

07 / Retrieval System

三層混合檢索機制

第 1 層：向量檢索
語義相似度搜尋，找出概念相關的記憶
→
第 2 層：圖譜實體搜尋
在知識圖譜中查找相關實體及其關係
→
第 3 層：區塊擴展
擴展檢索範圍，補充周邊上下文資訊

三層協作 = 精準定位 + 關聯發現 + 上下文補全，確保檢索結果既準確又完整
